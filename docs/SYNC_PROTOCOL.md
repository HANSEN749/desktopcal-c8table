# Cross-Device Sync Protocol

## Purpose

This protocol describes how DesktopCal desktop and the Android companion app exchange event data
through c8table while preserving local attachment storage on each device.

The protocol is intentionally schema-first. Desktop TypeScript types, Android Kotlin models, and
c8table fields must all map to the same `desktopcal.entry.v1` contract.

## Source Of Truth

c8table is the source of truth for event records. Each event maps to one c8table record.

Local databases are caches:

- Desktop uses IndexedDB for attachment blobs and local runtime state.
- Android will use Room for event cache and local media metadata.

## Entry Contract

See `packages/protocol/schemas/entry.v1.schema.json`.

Required core fields:

- `schema`: fixed string `desktopcal.entry.v1`.
- `localId`: stable UUID generated by the creating client.
- `title`.
- `date`: local calendar date, `YYYY-MM-DD`, never shifted by timezone conversion.
- `unit`.
- `kind`: `event` or `duration`.
- `importance`: 1-5.
- `completed`: optional boolean; absent means `false`.
- `attachments`.
- `createdAt`.
- `updatedAt`.

Optional mobile extension fields:

- `captureSource`: `desktop`, `android`, `import`, or `unknown`.
- `deviceId`: stable per-device id, not a user secret.
- `syncState`: local-only state, not required in c8table.
- `reminderAt`: ISO timestamp for notification scheduling.
- `location`: optional latitude/longitude metadata.
- `voiceNote`: optional local voice attachment pointer.

## Date And Time Rules

- `date` is a calendar key, not a timestamp.
- `date` must be written and read as the literal `YYYY-MM-DD` chosen by the user.
- `time` is optional `HH:mm` local wall-clock time.
- `reminderAt` may be a timestamp, but it is derived from `date` and `time` per device timezone.

## Unit And Marker Rules

Clients store semantic `unit` values. Shape is presentation metadata:

- `work` defaults to triangle.
- `research` defaults to circle.
- `review` defaults to square.
- `personal` defaults to diamond.
- `other` defaults to hexagon.

Clients store semantic `kind` values. Fill is presentation metadata:

- `event`: hollow marker.
- `duration`: solid marker.

## Sync Algorithm

### Pull

1. Read c8table records using `fieldKeyType=name`.
2. Parse structured fields into entry objects.
3. Preserve unknown fields for forward compatibility where possible.
4. Upsert records into local cache by remote record id or `localId`.
5. Recompute local reminders from entry state.

### Push Create

1. Create local entry with UUID `localId`.
2. Write structured c8table fields.
3. Store returned remote record id in local cache.

### Push Update

1. Compare local `updatedAt` with cached remote `updatedAt`.
2. If no remote change is known, PATCH structured fields.
3. If both sides changed, keep local change pending and surface a conflict state.

### Delete

Hard delete is allowed only after user confirmation. Clients should remove local attachment blobs
that are known to belong only to the deleted entry.

## Attachment Rules

Current attachment storage is local-first:

- Desktop local blobs use IndexedDB keys.
- Android local blobs use app-private file/content URIs.
- c8table records store attachment metadata only.

Attachment metadata must include:

- `id`.
- `storage`: `local` or `teable`.
- `name`.
- `mime`.
- `size`.
- `createdAt`.

Device-local pointer fields are optional and client-specific:

- `localBlobKey` for desktop.
- `androidLocalUri` for Android.
- `teableAttachmentId` for future remote storage.

## Security Rules

- API tokens stay local to each device.
- Standalone web OAuth uses Teable/c8table OAuth 2.0 PKCE. Browser clients store access and refresh
  tokens locally and never store a client secret.
- Tokens must not be written into git-tracked files.
- Device ids are identifiers, not authentication secrets.
- Recovery codes, GitHub tokens, and c8table tokens are outside the sync protocol.
